# UX-010: King mode — cascade death visual feedback

**Priority**: P1 | **Status**: TODO | **Tags**: `NEEDS-INPUT`

## Problem

When a king dies in Long Live the King, the entire team is eliminated (cascade death). This is the most dramatic event in the mode, but:

- **Dashboard**: Shows zero visual change. "ALIVE: 4/4" persists. The only indicator is a tiny footer message "Player #1 PlayerA eliminated!" that's unreadable from 3-5m.
- **Phone (cascade-killed player)**: Shows the exact same active-round view (team color + number + crown). Zero indication they are dead. They would keep trying to hold still, not realizing they're already eliminated.

## Affected modes

long-live-the-king only

## Audit references

- Summary: cross-mode issue #9, top priority fix #7
- King: `11_dash_king_died.png` — R-D03, R-D02, R-B02
- King: `12_phoneA_cascade_dead.png` — R-P02, R-B02

## Open question (NEEDS-INPUT)

**Phone cascade death**: What should the cascade-killed player see?
1. Standard dead screen (skull + "ELIMINATED") with added context: "Your king died — team eliminated"
2. Special cascade screen with team wipe messaging: "KING FALLEN — Red Team eliminated"
3. Same dead screen but with a brief dramatic transition (screen crack, flash to red, then dead screen)

**Dashboard cascade death**: What should spectators see?
1. Team panel grays out entirely + "RED TEAM ELIMINATED" banner overlay
2. Individual player cards in the team get skull overlays + team header changes to "ELIMINATED"
3. Brief animation (team panel flashes, shrinks, or fades) then settles to gray state

## Implementation plan (after input)

1. **Server**: The cascade death event already happens (`player:death` events fire for each team member). Verify the client receives them.
2. **Client phone**: When a `player:death` event arrives for the current player with a cascade flag, transition to the dead screen immediately. Add cascade context text.
3. **Client dashboard**: When multiple deaths arrive simultaneously (cascade), trigger the team-wipe visual. Gray out the team panel, show overlay text.
4. **Shared types**: May need a `cascadeDeath: boolean` flag on the `player:death` payload to distinguish cascade from normal death.
5. **Tests**: Server test that cascade deaths fire correctly. Client test that cascade dead screen renders. Dashboard test for team-wipe visual.
6. **Screenshot**: Capture the moment after a king dies.

## Dependencies

None, but UX-011 (crown on dashboard) is related.

## Estimated scope

Medium — server payload change + client rendering for both phone and dashboard. The server logic already handles cascade kills; this is about surfacing them visually.
