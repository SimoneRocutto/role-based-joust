# UX-010: King mode — cascade death visual feedback

**Priority**: P1 | **Status**: TODO

## Decision (resolved)

**Keep it simple: instant death for all team members, no special cascade animation.** The UI should show normal death state (skull on dashboard cards, dead screen on phone). No dramatic transitions or special "king fell" banners. Aesthetics can be revisited later when the entire UI gets a design pass.

## Problem

When a king dies in Long Live the King, the entire team is eliminated (cascade death). Currently:

- **Dashboard**: Team member cards don't immediately reflect death state. The alive counter may lag.
- **Phone (cascade-killed player)**: May not transition to dead screen properly.

## Affected modes

long-live-the-king only

## Implementation plan

1. **Verify server**: Cascade death events (`player:death`) already fire for each team member when the king dies. Confirm these propagate to the client.
2. **Client phone**: Ensure `player:death` events for cascade-killed players transition the phone to the standard dead screen immediately.
3. **Client dashboard**: Ensure all team member cards show skull/dead state when cascade deaths arrive (should already work if `game:tick` updates are fast enough).
4. **Tests**: Verify cascade death events reach the client and trigger correct UI state changes.
5. **Screenshot**: Capture dashboard + phone state after a king dies to confirm dead state renders.

## Dependencies

None, but UX-011 (crown on dashboard) is related.

## Estimated scope

Medium — server payload change + client rendering for both phone and dashboard. The server logic already handles cascade kills; this is about surfacing them visually.
