# UX-002: Phone active — add HP bar

**Priority**: P0 | **Status**: TODO | **Tags**: `NEEDS-INPUT`

## Problem

No HP information is visible at a glance on the active phone screen in any mode. The "100%" text in the top-right corner (where present) is too small for chest-mounted play. Players cannot answer "how much health do I have?" with a one-second glance.

## Affected modes

All 7 modes

## Audit references

- Summary: cross-mode issue #1, top priority fix #1
- Every mode's active phone screenshot flags R-P01

## Open question (NEEDS-INPUT)

Visual style options for HP display:

1. **Bar along screen edge** — thin horizontal bar at top or bottom that depletes left-to-right. Minimal, doesn't compete with other content.
2. **Background color gradient** — the full-screen background shifts from green (full HP) to yellow to red (low HP). Bold, visceral, zero-glance readable. NOTE: audit reports suggest this may already be partially implemented (green = alive) but it's not confirmed whether it transitions to red as HP drops.
3. **Numeric display** — large "87%" or "HP: 87" somewhere prominent on screen.
4. **Combination** — e.g., background color shift + thin bar for precision.

The user should pick which approach fits the party-game aesthetic. The background color gradient is the audit's recommended approach (highest visibility for chest-mounted play at party distances).

## Implementation plan (after input)

1. **Client**: Add HP visualization to `PlayerView` active state
   - The `game:tick` socket event already sends per-player HP data to the client
   - The Zustand store already has `hp` in the player state
   - Implementation depends on chosen style (CSS background transition vs. bar component)
2. **Tests**: Client unit test that HP component renders and updates. Test edge cases (0%, 100%, rapid changes).
3. **Screenshot + audit**: Capture at full HP and damaged states to verify readability.

## Dependencies

None — can be done independently. Best done alongside UX-001 since both modify the active phone layout.

## Estimated scope

Small-medium — client-only change. The data pipeline already exists (server sends HP via `game:tick`). This is purely a rendering task.
