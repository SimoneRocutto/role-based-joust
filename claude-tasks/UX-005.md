# UX-005: Dashboard — round-end summary overlay

**Priority**: P1 | **Status**: TODO | **Tags**: `NEEDS-INPUT`

## Problem

When a round ends, no "ROUND OVER" banner, winner highlight, or standing summary appears on the dashboard in 5/7 modes. The round-end state looks like the active state. Exception: classic-teams round-end dashboard is excellent and should be used as the template.

## Affected modes

classic, role-based, death-count, death-count-teams, domination (5/7)

Classic-teams already has a good round-end screen ("ROUND 1 COMPLETE" + medal-ranked team scores + nested individual breakdowns). Long-live-the-king's game-end screen is also decent.

## Open question (NEEDS-INPUT)

Options for the round-end dashboard:

1. **Minimal banner** — overlay "ROUND OVER" in large text on the existing layout. Quick to implement, limited info.
2. **Full summary (classic-teams style)** — replace the player grid with a ranked leaderboard showing: round winner, per-player scores, team scores (if applicable), "NEXT ROUND" button. This is the gold standard from the audit.
3. **Hybrid** — keep the player grid visible but add a summary panel above or below it.

The classic-teams round-end screen (`16_dash_round_end.png`) should be examined as the reference implementation. The question is whether all modes should adopt this exact layout or whether non-team modes need a simpler version.

## Implementation plan (after input)

1. **Client**: Create a `RoundEndOverlay` or `RoundEndSummary` component for the dashboard
   - Show when `gamePhase === 'round-ended'` or `gamePhase === 'finished'`
   - Content varies by mode (team vs individual, death-count vs survival)
   - The `round:end` socket event already sends score data
2. **Examine classic-teams implementation** — it already works. Understand its component structure and generalize it.
3. **Tests**: Unit test the overlay renders with correct data for each mode type.
4. **Screenshot + audit**: Capture round-end state for all modes.

## Dependencies

None strictly, but pairs well with UX-004 (phase differentiation).

## Estimated scope

Medium-large — needs a new component, mode-aware content logic, and testing across all modes. Classic-teams existing implementation reduces the work.
